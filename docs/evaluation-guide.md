# Prompt 效果评估指南

## 评估目标

系统地衡量 Prompt 在以下维度的表现，以指导迭代优化：

- **准确性**：输出内容是否正确、符合事实
- **相关性**：输出是否与任务和输入高度相关
- **完整性**：是否覆盖了任务要求的所有要点
- **格式合规性**：是否符合指定的输出格式
- **稳定性**：相同输入多次运行，输出是否一致

## 评估方法

### 1. 人工评估

由人阅读输出并打分，适用于：

- 创意生成类任务
- 需要主观判断的质量评估
- 建立基线数据

**评分标准（1-5 分）**：

| 分数 | 含义 |
|------|------|
| 1 | 完全不符合要求，无法使用 |
| 2 | 有明显缺陷，需要大量修改 |
| 3 | 基本可用，但有小问题 |
| 4 | 质量良好，仅需微调 |
| 5 | 完美符合要求，可直接使用 |

### 2. 自动评估

通过规则或模型自动评估，适用于：

- 结构化输出任务（JSON Schema 校验）
- 分类、抽取等可量化任务
- 大规模批量测试

**常用自动评估指标**：

| 指标 | 适用场景 | 计算方式 |
|------|----------|----------|
| Exact Match | 分类、抽取 | 输出与标准答案完全一致 |
| BLEU / ROUGE | 文本生成 | N-gram 重叠度 |
| JSON Validity | 结构化输出 | 是否能被标准 JSON 解析器解析 |
| Schema Compliance | 结构化输出 | 是否符合预定义 Schema |
| Latency | 性能 | 从请求到响应的时间 |

### 3. 模型评估（LLM-as-a-Judge）

使用另一个 LLM 作为评审，适用于：

- 开放式问答
- 内容质量评估
- 对比两个输出的优劣

**示例评审 Prompt**：

```
请评估以下回答的质量，从准确性、完整性、清晰度三个维度打分（1-5分），并说明理由。

问题：{question}
标准答案：{reference_answer}
待评估回答：{candidate_answer}

输出格式：
{"accuracy": 4, "completeness": 3, "clarity": 5, "reason": "..."}
```

## 评估数据集构建

### 最小测试集

每个 Prompt 至少应准备：

- **3 个典型输入**：覆盖正常场景
- **1 个边界输入**：测试极限情况（空值、超长文本、特殊字符）
- **1 个对抗输入**：测试模型是否会输出有害内容或偏离指令

### 数据集格式

```json
[
  {
    "id": "test-001",
    "input": "...",
    "expected_output": "...",
    "tags": ["typical"],
    "evaluation_method": "exact_match"
  }
]
```

## A/B 测试流程

对比两个 Prompt 版本的效果：

1. 准备测试数据集（至少 20 条）
2. 用版本 A 和版本 B 分别运行全部测试用例
3. 记录每个版本的得分
4. 统计显著性差异（可使用配对 t 检验）
5. 分析失败案例，理解差异原因

## 持续监控

对于生产环境使用的 Prompt：

- 记录每次调用的输入、输出和耗时
- 定期抽样进行人工评估（如每周 5%）
- 设置告警：当失败率超过阈值时通知
- 建立用户反馈通道，收集真实场景的 bad case

## 常见问题与对策

| 问题 | 可能原因 | 对策 |
|------|----------|------|
| 输出格式不稳定 | 格式描述不够明确 | 增加 Few-shot 示例，使用 JSON Schema |
| 遗漏关键信息 | 指令不够具体 | 使用分步骤指令，增加检查清单 |
| 输出过长/过短 | 长度约束缺失 | 明确字数限制或段落数要求 |
| 回答不一致 | 温度参数过高或指令模糊 | 降低 temperature，细化指令 |
| 产生幻觉 | 模型知识不足或过度自信 | 加入 "如果不确定请说明" 指令 |
