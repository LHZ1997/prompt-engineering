# Few-Shot Prompting（少样本提示）

## 核心思想

通过在 Prompt 中提供少量输入-输出示例（demonstrations），让模型学习任务模式、输出格式和隐含规则，从而在新输入上产生期望的输出。

## 适用场景

- 需要特定输出格式的任务（如 JSON、表格、特定文风）
- 分类、标注、抽取等模式识别任务
- 风格迁移、语气控制
- 复杂指令难以用语言精确描述时

## 基本模式

```
以下是几个示例：

输入：{example_input_1}
输出：{example_output_1}

输入：{example_input_2}
输出：{example_output_2}

输入：{actual_input}
输出：
```

## 最佳实践

### 1. 示例数量

- 简单任务：1-3 个示例即可
- 复杂任务：3-5 个示例，覆盖主要变体
- 示例过多会占用上下文窗口，建议控制在总长度的 30% 以内

### 2. 示例质量

- **代表性**：覆盖常见情况和边界情况
- **一致性**：示例之间遵循相同的规则和格式
- **准确性**：确保示例输出完全正确，模型会模仿错误

### 3. 示例排列

- 将最相关的示例放在靠近实际输入的位置
- 对于分类任务，各类别示例数量尽量均衡

### 4. 动态 Few-Shot

对于大量示例场景，可使用向量检索（RAG）动态选取与当前输入最相似的 K 个示例，而非静态写入 Prompt。

## 变体：Zero-Shot vs One-Shot vs Few-Shot

| 类型 | 示例数 | 适用情况 |
|------|--------|----------|
| Zero-Shot | 0 | 任务简单、指令清晰、模型已具备能力 |
| One-Shot | 1 | 仅需说明输出格式 |
| Few-Shot | 2-5 | 需要学习复杂模式或特定规则 |

## 常见陷阱

- **示例偏见**：示例覆盖不全面导致模型泛化能力差
- **格式不一致**：示例输出格式不统一，导致模型输出混乱
- **标签泄漏**：示例中无意间包含了测试数据的模式
